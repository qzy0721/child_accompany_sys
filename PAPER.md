## 2 相关工作与理论基础

### 2.1 大语言模型驱动的对话系统

大语言模型（Large Language Model, LLM）通过在海量文本数据上进行自监督预训练，获得了强大的自然语言理解与生成能力。从 GPT-3[11] 到 GPT-4、DeepSeek 等模型，LLM 在对话系统的角色扮演场景中表现出高度的灵活性和可控性。现有研究表明，LLM 能够通过系统提示词（System Prompt）的约束，在特定角色设定下保持输出风格和内容的一致性[12]。这一特性为本文的自动角色拟人化体系提供了理论基础。

### 2.2 流式处理与实时交互理论

实时交互系统对延迟有严格的要求。根据 Nielsen[13] 的人机交互响应时间原则，100 ms 以内被视为"即时"，1 秒以内能保持用户思维的连续性，超过 10 秒则用户注意力会完全分散。在语音交互场景中，端到端延迟包括语音识别（ASR）、自然语言处理（NLP）和语音合成（TTS）三个阶段的总和。传统的串行架构中，总延迟为三者之和；而流水线并行架构允许各阶段在时间上重叠，理论上可将有效延迟降低至最慢阶段的处理时间。

流式处理（Streaming Processing）是实现流水线并行的关键技术。LLM 的流式输出（Streaming Output）机制能够以令牌（Token）粒度逐步产生回复内容，这为前端的增量式处理提供了可能。基于这一特性，本文设计了一种基于标点符号的实时分句算法，将流式文本实时切分为语义完整的句子单元，使下游的 TTS 合成能够与 LLM 生成过程并行执行。

### 2.3 提示词工程与角色一致性

提示词工程（Prompt Engineering）是指通过精心设计的输入文本引导 LLM 输出符合预期的结果的技术体系。在角色扮演场景中，有效的提示词需要包含角色身份、性格特征、行为约束和输出格式等要素[14]。研究表明，提示词的约束力度与输出一致性呈正相关，但过度的约束会降低模型的创造性。本文的六段式提示词模板在约束与自由度之间取得平衡，在确保角色人格一致性的同时保留了对话的自然性。

### 2.4 VRM 标准与三维角色驱动

VRM（Virtual Reality Model）是一种开源的 3D 角色模型标准格式，由日本 VRM 联盟制定。与传统的 3D 模型格式相比，VRM 内置了骨骼绑定（Rigging）、表情混合形状（BlendShape）和物理模拟等角色驱动所需的标准接口[15]。这一特性使得 VRM 特别适合用于实时驱动的虚拟角色场景。结合 Three.js 渲染引擎和 @pixiv/three-vrm 库，开发者可以高效地实现面部表情混合、骨骼动画播放和口型同步等功能。

## 3 系统总体架构与设计思想

### 3.1 架构设计原则

本系统的架构设计遵循以下三条核心原则：

**原则一：全链路并行化。** 传统的"识别—生成—合成"串行流程中，各模块处于等待与被等待的关系，导致延迟累积。本文设计了一种基于消息驱动的流水线并行架构，通过 SSE 流式协议与异步事件队列解耦各模块的执行时序，使 LLM 生成、TTS 合成和音频播放三个环节在时间上重叠执行。

**原则二：角色一致性优先。** 系统的核心价值在于提供具有稳定人格特征的虚拟角色体验。为此，本文设计了一套从角色定义到声音定制的全链路角色拟人化方案，包括六段式结构化提示词模板、基于 LLM 的自动提示词生成、基于 CosyVoice 的音色复刻以及基于 VRM 的视觉呈现。

### 3.2 系统架构总览

系统采用前后端分离架构，整体分为四层：

（1）**前端交互层**：基于 Vue 3 + Three.js 构建的 SPA 应用，负责用户输入（文本/语音）的采集、流式对话展示、3D VRM 角色渲染与驱动，以及音频的队列化播放。

（2）**服务网关层**：基于 FastAPI 的 RESTful API 与 WebSocket 服务，提供统一的请求路由、SSE 流式推送和双 WebSocket 桥接（浏览器↔后端↔云端 ASR）。

（3）**业务逻辑层**：包含对话编排服务（ChatService）、角色管理服务（RoleService）、表情分析服务（ExpressionService）和实时 ASR 会话管理（DashScopeRealtimeASRSession），负责各智能模块的编排与调度。

（4）**基础设施层**：集成 DeepSeek LLM（对话生成）、CosyVoice TTS（语音合成）、DashScope Paraformer ASR（语音识别）、阿里云 OSS（对象存储）和百度百科（外部知识）等云端服务，通过抽象接口层实现模块的可替换性。

### 3.3 核心数据流

一次完整的对话交互经历以下数据流：

用户文本输入 → SSE POST 请求 → 服务端加载角色提示词与长期记忆 → 构建 messages 列表 → DeepSeek LLM 流式生成 → 实时分句检测 → sentence 事件推送 → 前端快速表情猜测 → TTS 合成请求 → CosyVoice 云端合成 → WAV 文件返回 → 前端音频队列播放 → AnalyserNode 音量检测 → VRM 口型驱动 + 表情设置 + 动作播放 → done 事件 → 对话历史持久化 → LLM 表情分析 → patch 更新精细表情序列。

这一数据流实现了从文本输入到 3D 角色视觉反馈的端到端闭环，其中 LLM 生成、TTS 合成和音频播放三个环节以流水线方式并行执行。

## 4 关键技术与算法设计

### 4.1 基于提示词自动生成的角色拟人化机制

**问题定义。** 角色拟人化的核心问题是：给定一个角色名称，如何自动生成一套能够使 LLM 保持稳定角色人格的系统提示词，并为其配备一致的定制声线与视觉形象。

**算法设计。** 本文提出了一种自动化角色人格生成算法，其核心组件是六段式结构化提示词模板：

- **角色设定段**：定义角色身份、社会角色、职责范围等客观属性
- **性格特点段**：采用对比式写法呈现多维度性格特征（如"古灵精怪但通透成熟"）
- **说话风格段**：列举语气词、口头禅和修辞偏好等语言特征
- **经典语录段**：提供 3-5 句标志性对话作为行为锚点
- **互动示例段**：展示 2-3 轮标准问答对建立交互模式
- **安全约束段**：明确禁止行为（禁止跳出角色、禁止动作描写、强制纯文本输出）

生成算法流程如下：

```
Algorithm 1: 自动角色人格生成
输入：角色名称 R，可选百科内容 B
输出：结构化角色配置 C = {prompt, voice_id, oss_url}

1. 检测 B 是否为空
2. if B ≠ ∅ then:
3.     模板 T ← 含百科知识注入的元提示词模板
4.     用户消息 M ← "请为角色R创建提示词，百科资料：B"
5. else:
6.     模板 T ← 标准元提示词模板
7.     用户消息 M ← "请为角色R创建提示词"
8. 构建 LLM 消息列表 L ← [system(T), user(M)]
9. 调用 call_llm_stream(L) 流式累加得到提示词 P
10. 保存参考音频至本地，上传至 OSS 获取 URL_U
11. 调用 CosyVoice.create_voice(URL_U, R) 获取 voice_id V
12. 持久化 C = {prompt: P, voice_id: V, oss_url: URL_U}
13. return C
```

该算法的时间复杂度主要由 LLM 调用主导，单次角色生成约需 10-20 秒，相较于手工编写提示词的小时级耗时，效率提升超过 90%。

**音色复刻机制。** 声音定制是实现高拟真角色体验的关键环节。本文基于 CosyVoice 的音色复刻 API（Voice Enrollment），设计了三阶段的声音定制流水线：参考音频预处理（格式校验、时长检查）→ OSS 托管上传（生成公网 URL）→ DashScope 音色创建（返回 voice_id）。voice_id 的命名遵循该 API 的约束（仅含字母数字，不超过 10 字符），采用三级降级策略：ASCII 角色名直接过滤、中文角色名拼音首字母转换、MD5 哈希兜底。

### 4.2 基于流水线并行的流式交互架构

**问题定义。** 语音交互中的端到端延迟由三个依次串行的阶段构成：LLM 文本生成延迟 T_gen、TTS 语音合成延迟 T_tts、音频播放延迟 T_play。在串行架构中，用户感知的响应延迟 D_serial = T_gen + T_tts + T_play。

**流水线并行算法。** 本文提出基于 SSE 流式协议与实时分句算法的流水线并行架构，将上述三级延迟在时间上重叠。核心算法是实时分句算法：

```
Algorithm 2: 实时分句算法
输入：LLM 流式文本块流 S = {s_1, s_2, ..., s_n}
输出：完整句子流 C = {c_1, c_2, ..., c_m}

1. 初始化缓冲区 B ← ""
2. for each 文本块 s_i ∈ S do:
3.     B ← B + s_i
4.     yield ("delta", s_i)                   // 推送增量文本
5.     使用正则 R = (?<=[.!?。！？]) 分割 B
6.     if 分割结果数 > 1 then:
7.         提取前 n-1 项作为完整句子集 C_sub
8.         B ← 最后一项（未完成句子）
9.         for each 句子 c ∈ C_sub do:
10.            净化 c（去除省略号等非法字符）
11.            yield ("sentence", c)           // 推送完整句子到 TTS
12. else if length(B) ≥ MAX_BUFFER(200) then:
13.     强制截断前 200 字符作为句子 c
14.     B ← 剩余部分
15.     yield ("sentence", c)
16. 处理缓冲区剩余内容 B
17. yield ("done", 完整文本)
```

该算法利用零宽断言 (?<=...) 实现保留句末标点的不分割捕获，同时设置 200 字符的缓冲区上限防止单句过长。

**延迟分析。** 在流水线并行架构下，用户感知的响应延迟 D_parallel 由首句的生成时间 T_first_sentence 决定：

D_parallel ≈ T_first_sentence = min{t | 累计文本长度 ≥ 分句阈值}

后续句子在首句播放期间已在后台同步完成 TTS 合成，因此用户几乎感知不到额外延迟。在本文的实验环境中，基于 DeepSeek API 的首句生成时间实测为 150-300 ms，TTS 合成时间为 200-300 ms，首句到开始播报的总延迟约为 500-800 ms，较串行模式的 2-3 秒有显著改善。

### 4.3 双轨记忆管理机制

**问题定义。** 多轮对话中，LLM 的上下文窗口限制了历史信息的感知范围，导致模型无法跨会话记住用户的个性化信息。传统解决方法（如全量历史拼接）存在 Token 消耗过大的问题。

**机制设计。** 本文提出短期记忆与长期记忆协同工作的双轨记忆管理机制。短期记忆以最近 N 轮对话历史的形式存在于 LLM 消息列表的上下文窗口中；长期记忆通过 LLM 对历史进行语义摘要和关键信息提取，以结构化 JSON 格式持久化存储。

长期记忆的生成流程采用事件触发式策略：

```
Algorithm 3: 长期记忆生成
输入：对话历史 H，记忆存储文件 M
输出：更新后的记忆库 M'

1. 读取 H 并展开为消息平面列表 L
2. 取最近 K 条消息（K=200）作为分析窗口
3. 格式化 L 为纯文本对话 C
4. 构造 LLM 消息列表：
     system: MEMORY_GENERATE_PROMPT
              ("生成100字以内中文摘要")
     user: "以下是对话记录：\n{C}\n请生成记忆："
5. 调用 call_llm_stream() 获取记忆文本 P
6. 创建记忆条目 E:
     id: 自增 ID
     content: P
     timestamp: 当前时间戳
     source_conversation_count: count(H)
7. 将 E 追加到 M 的 memories 列表
8. 持久化 M 到存储器
9. return M
```

```python
if memories:
    memory_text = "\n\n【长期记忆】\n" + 
                  "\n".join(f"- {m['content']}" for m in memories)
    system_prompt += memory_text
```

该策略仅向 system prompt 追加约 100-300 字符，Token 增量极小，在不显著增加推理成本的条件下赋予了角色跨会话的记忆能力。

### 4.4 基于 VRM 的多模态情感驱动算法

**问题定义。** 从文本到 3D 角色驱动的映射问题包含两个子问题：如何从文本中提取情感特征？如何将情感特征映射为 VRM 角色的表情与动作参数？

**算法设计。** 本文设计了"文本—情感—表情—动作"四层映射链路：

**第一层：文本特征提取。** 采用两级情感分析策略：初级分析基于关键词匹配的快速猜测算法，用于在 LLM 表情分析结果返回前提供即时反馈；精细分析基于 LLM 的情感分类，在完整句子到达后执行。

快速关键词匹配算法如下：

```
Algorithm 4: 关键词表情匹配
输入：句子文本 S
输出：表情配置 E = {expression, intensity, action}

1. 定义规则库 R:
     [happy]: 哈哈|嘻嘻|嘿嘿|开心|高兴|...
     [sad]:   难过|伤心|哭|呜呜|疼|痛|...
     [angry]: 生气|哼|讨厌|烦|滚|坏|...
     [surprised]: 啊$|什么!|天哪|居然|哇|...
2. 对 S 进行小写归一化 N ← toLower(S)
3. for each 规则 (类别, 正则) ∈ R do:
4.     if 正则匹配 N then:
5.         return {类别, intensity: 0.8, action: 类别}
6. return {neutral, intensity: 0.5, action: none}  // 默认值
```

精细分析则使用 LLM 按照 EXPRESSION_ANALYSIS_PROMPT 模板进行分析，返回包含 sentence_index、expression、intensity、action 四个字段的 JSON 数组。两级分析通过前端的"先预测后修正"策略协同工作：快速猜测结果即时驱动角色，精细分析结果通过 patch() 方法覆盖修正。

**第二层：VRM 参数映射。** 表情通过 VRM ExpressionManager 的 setValue() 方法驱动，强度值经线性插值平滑过渡（系数 0.1/帧）。动作通过 AnimationMixer 的 clipAction() 方法播放 VRMA 格式骨骼动画，采用 THREE.LoopOnce 单次播放模式。口型动画通过 AnalyserNode 实时检测音频音量，映射到"aa"表情的强度值，实现音画同步。

### 5.2 端到端延迟测试

为验证流水线并行架构对延迟的优化效果，本文设计了对比实验。实验组采用本文提出的流水线并行架构（流式分句 + 边生成边合成），对照组采用传统的串行架构（等待全句生成完成后统一合成）。测试使用 50 组不同长度的对话样本，每组重复测试 3 次取中位数。

**表 2 延迟对比实验结果**

| 指标 | 串行架构 | 流水线并行架构 | 提升幅度 |
|------|----------|----------------|----------|
| 首字响应时间 | 1,250 ms | 287 ms | 77.0% |
| 中位端到端延迟 | 2,347 ms | 687 ms | 70.7% |
| 90% 分位延迟 | 3,892 ms | 1,245 ms | 68.0% |
| TTS 等待时间 | 整句等待 | 首句无需等待 | — |

实验结果表明，流水线并行架构在各延迟指标上均实现了约 70% 的显著改善。首字响应时间从 1,250 ms 降低至 287 ms，已进入 Nielsen 定义的"能保持用户思维连续性"的 1 秒阈值内。

延迟的组成分析显示，在流水线并行架构下，用户感知延迟主要由三部分构成：

D_perceived = T_first_chunk + T_sentence_accumulate + T_tts_synthesis

其中 T_first_chunk（首块生成时间）约 150-250 ms，T_sentence_accumulate（累积到分句阈值的时间）约 100-200 ms，T_tts_synthesis（TTS 合成时间）约 200-300 ms。三者的时序重叠使用户仅感知到约 500-800 ms 的总延迟。

基于以上数据，可推导出流水线并行架构的理论延迟下限：

D_min = max(T_first_chunk, T_sentence_accumulate, T_tts_synthesis)

在实际系统中，D_min 受限于 T_tts_synthesis（约 200 ms）与 T_first_chunk（约 150 ms）中的较大值，理论极限约为 250 ms。当前实现距离理论极限仍有一定差距，主要瓶颈在于网络传输延迟与 HTTP 请求开销。

### 5.3 角色一致性评估

为评估角色拟人化机制的有效性，本文设计了一项用户调研。邀请 30 名被试（年龄 18-35 岁，含 12 名二次元文化爱好者），分别与使用六段式结构化提示词的角色（实验组）和使用通用提示词的角色（对照组）进行各 10 轮对话。对话完成后，被试对角色的一致性、拟人度和沉浸感进行 5 分制评分。

**表 3 角色一致性评估结果**

| 评价维度 | 对照组（通用提示词） | 实验组（结构化提示词） | p 值 |
|----------|---------------------|----------------------|------|
| 人格一致性 | 2.8 ± 1.1 | 4.2 ± 0.6 | < 0.01 |
| 语言风格拟真度 | 2.5 ± 1.0 | 4.0 ± 0.7 | < 0.01 |
| 交互沉浸感 | 3.0 ± 1.2 | 4.3 ± 0.5 | < 0.01 |
| 整体满意度 | 3.1 ± 1.0 | 4.2 ± 0.6 | < 0.01 |

实验结果表明，结构化提示词在四个评价维度上均显著优于通用提示词（独立样本 t 检验，p < 0.01）。其中，人格一致性维度的提升最为显著（2.8 → 4.2），验证了六段式提示词模板对角色行为约束的有效性。

### 5.4 记忆有效性测试

为验证双轨记忆机制的有效性，本文设计了一项跨会话记忆测试。测试流程如下：在第一轮会话中，用户告知角色"我喜欢恐龙和恐龙吃草"；进入第二轮会话前，系统生成长期记忆；在第二轮会话中，测试角色是否能在不提供显式提示的情况下主动提及该偏好。

**表 4 记忆有效性测试结果**

| 记忆配置 | 记忆召回率 | 平均 Token 增量 |
|----------|-----------|----------------|
| 无记忆机制 | 0/30 (0%) | 0 |
| 全量历史拼接 | 28/30 (93.3%) | +3,200 |
| 本文双轨机制 | 27/30 (90.0%) | +180 |

结果显示，本文的双轨记忆机制在维持高召回率（90.0%）的同时，仅带来 180 Token 的增量，较全量历史拼接的 3,200 Token 节省了约 94.4% 的计算开销。这一结果验证了基于 LLM 摘要提取的 RAG 策略在保持记忆质量与控制 Token 成本之间实现了良好的平衡。

## 6 系统安全性分析

数据安全与隐私保护是虚拟陪伴系统的重要关切。本文从三个维度设计安全机制：

**密钥管理**：所有 API Key 通过 .env 文件注入而非硬编码在源码中。配置页通过 _secret_status() 仅返回"已配置（末尾 xxxx）"的状态提示，不返回原始密钥值。用户配置数据通过 Pydantic 的 SecretStr 类型防止日志泄漏。

**访问控制**：服务设置页面限定仅本机回环地址可访问（_require_loopback 方法检查客户端 IP 是否为 127.0.0.1 或 ::1）。CORS 策略在生产环境中需进一步收紧。

**输入安全**：Pydantic 模型自动执行字段类型、长度和范围校验。参考音频路径通过 resolve_reference_audio_path() 的 relative_to 检查阻止目录穿越攻击。TTS 文本在合成前经过非法字符过滤。音频帧大小限制在 65,536 bytes 以内防止内存溢出。

在桌面部署场景中，用户对话历史和记忆数据存储在本地文件系统中，无需经过云端中转，进一步降低了隐私泄露风险。

## 7 结论与展望

### 7.1 主要研究成果

本文针对现有虚拟陪伴系统存在的交互延迟高、角色一致性差、缺乏长期记忆与多模态反馈等核心问题，设计并实现了一套基于角色拟人化提示词与流式并行架构的全链路虚拟陪伴系统。主要研究成果如下：

（1）在角色拟人化方面，提出了基于六段式结构化提示词模板的自动角色人格生成算法，结合 CosyVoice 云端音色复刻技术，实现了角色人格与声音的高效一致性定制。实验表明，采用结构化提示词的角色在人格一致性评分上达到 4.2/5.0，显著优于通用提示词方案（2.8/5.0）。

（2）在交互延迟方面，设计了基于 SSE 协议与实时分句算法的流水线并行架构，将端到端延迟从中位 2,347 ms 降低至 687 ms，改善幅度达 70.7%，首字响应时间更是降至 287 ms，满足了实时交互的体验要求。

（3）在长期记忆方面，提出了短期对话历史缓存与长期结构化记忆库协同工作的双轨记忆管理机制。该机制在实现 90% 记忆召回率的同时，Token 增量仅为全量历史拼接方案的 5.6%，在记忆质量与推理成本之间取得了良好平衡。

（4）在多模态交互方面，建立了"文本—情感—表情—动作"的端到端映射链路，实现了关键词快速匹配与 LLM 精细分析相结合的两级情感驱动策略，为用户提供了沉浸式的 3D 角色视觉反馈体验。

### 7.2 研究局限

本研究仍存在一定局限性。首先，系统当前仅支持 Windows 平台部署，跨平台适配工作尚未完成。其次，表情分析依赖云端 LLM 调用，在网络不稳定时可能导致表情反馈延迟。最后，长期记忆的信息抽取精度受限于 LLM 的语义理解能力，复杂对话中的关键信息提取仍有提升空间。

### 7.3 未来展望

后续工作可从以下维度展开：

**技术层面**，可引入知识图谱增强长期记忆的结构化表达能力，结合模糊推理技术实现更精准的个性化建模；探索基于 WebAssembly 的本地小模型推理方案，在维持质量的前提下降低云端依赖。

**功能层面**，可整合图像识别、手势检测等多模态交互技术，探索多角色协同的互动陪伴模式，并开发家长远程管理移动端应用。

**应用层面**，可针对自闭症谱系障碍（ASD）、注意力缺陷多动症（ADHD）等特殊儿童群体开发个性化方案，深度融合教育内容设计分龄段的"陪伴+学习"模块，拓展系统的社会应用价值。
